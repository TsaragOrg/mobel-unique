## Goal

Deliver durable catalog image variants so dense admin render review and public
catalog cards load bounded preview bytes, while admin large previews and public
sofa detail pages can still inspect the accepted original render.

## Architecture

Use stored variants as the product contract. Each active catalog render image
that can be shown in admin render review or public catalog reads keeps its
existing `storage_assets` row as the original, and receives `small` and
`medium` variant `storage_assets` rows linked through a new
`storage_asset_variants` table.

This plan does not use Supabase Storage Image Transformations for serving or
generation. Variant bytes are generated by first-party server code with the
same deterministic local behavior in tests, local Supabase, DEV, and PROD. This
avoids adding Supabase transformation plan, billing, cache, and local fallback
dependencies to the rollout.

Preview variants use longest-edge resizing with no crop:

- `small`: longest edge at most 320 px.
- `medium`: longest edge at most 1280 px.
- `original`: the existing canonical accepted asset, with no extra resizing.

Generated preview variants should be JPEG with a stable quality setting unless
the decoded image contains alpha, in which case use PNG to preserve the visual
result. Variant object paths include immutable variant asset ids so public CDN
and browser caches cannot confuse old and new images.

## Scope

This plan includes:

- a durable variant relationship table and indexes;
- first-party variant generation helpers for the Next.js admin API and the
  fabric render Edge Function;
- admin upload completion variants for source photos and manual renders before
  those assets become usable;
- fabric render worker candidate variants before a candidate is marked
  preview-ready;
- variant-aware admin preview reads through the existing protected admin route;
- publication of public original copies and public `small`/`medium` variants in
  one safe publish workflow;
- public catalog API medium image URLs and public sofa detail original image
  URLs;
- an idempotent backfill script for existing private and public catalog render
  assets;
- local seed and DEV catalog snapshot compatibility;
- focused tests, smoke/source checks, roadmaps, and manual QA notes.

This plan does not include:

- using raw Supabase transformation URLs in browser code;
- making `catalog-private-assets` public;
- changing accepted upload limits or client-side resize behavior;
- changing fabric swatch crop behavior;
- changing AI provider prompts or generation quality;
- adding customer simulation image variants;
- physically purging every historical unreachable public original object that
  existed before this plan. Public read paths and active variant rows are the
  required rollout boundary.

## Data Shape

Add this logical table:

```sql
create table public.storage_asset_variants (
  original_asset_id uuid not null references public.storage_assets (id) on delete restrict,
  variant_kind text not null,
  variant_asset_id uuid not null references public.storage_assets (id) on delete restrict,
  generation_kind text not null default 'stored',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  primary key (original_asset_id, variant_kind),
  constraint storage_asset_variants_kind_check
    check (variant_kind in ('small', 'medium')),
  constraint storage_asset_variants_generation_kind_check
    check (generation_kind in ('stored')),
  constraint storage_asset_variants_not_self_check
    check (original_asset_id <> variant_asset_id)
);
```

Required indexes:

- unique index on `variant_asset_id`;
- lookup index on `(original_asset_id, variant_kind)`;
- lookup index on `(variant_asset_id)`.

The original asset is not duplicated into this table as `original`. API code
resolves `original` directly from `storage_assets.id`. API code resolves
`small` and `medium` through `storage_asset_variants`.

Variant asset rows must use the same bucket and visibility as their original
asset. Private variants stay in `catalog-private-assets`; public variants stay
in `catalog-public-assets`.

Suggested object paths:

```text
variants/{original_asset_id}/small/{variant_asset_id}.jpg
variants/{original_asset_id}/medium/{variant_asset_id}.jpg
variants/{original_asset_id}/small/{variant_asset_id}.png
variants/{original_asset_id}/medium/{variant_asset_id}.png
```

The extension follows the actual generated content type.

dimensions and content type metadata from the same active `storage_assets` row
that owns the served object.

## Lifecycle Rules

Variant reads are valid only when both sides are active:

- `storage_assets.lifecycle_state = 'active'` for the original asset;
- `storage_assets.lifecycle_state = 'active'` for the variant asset;
- `storage_asset_variants` points from that original to that variant kind;
- private read paths keep private variants behind the admin preview endpoint;
- public read paths expose only public variants in `catalog-public-assets`.

When an original asset is deleted, deactivated, or purged, implementation must
call `deactivate_storage_asset_variants(original_asset_id)` directly or through
the `storage_assets` lifecycle trigger. That helper marks linked variant asset
rows unavailable for read paths and records matching lifecycle timestamps where
the original lifecycle operation provides them.

Admin preview and public read code must not trust an old variant row by itself.
They must join back to the original asset and reject the read when either the
original or variant asset is not active.

## Admin Preview API Shape

Use the existing protected route with an additive selector:

```text
GET /api/admin/storage-assets/{asset_id}/preview?variant=small
GET /api/admin/storage-assets/{asset_id}/preview?variant=medium
GET /api/admin/storage-assets/{asset_id}/preview?variant=original
```

Rules:

- default missing `variant` to `original` for route compatibility;
- reject unsupported variant values with a safe `400` error;
- return safe `404` for missing `small` or `medium` rows;
- never fall back from `small` or `medium` to original bytes;
- keep `Cache-Control: no-store`;
- never return private bucket paths or signed Supabase URLs.

## Rollout Notes

Rollout order matters:

1. Deploy the schema and code that can create and read variants.
2. Run the backfill in DEV and validate counts.
3. Switch admin and public UI reads to variant-aware fields.
4. Run the backfill in PROD before relying on medium catalog URLs for public
   cards.
5. Verify publication in the target environment creates public variants for a
   newly published sofa.

If a deployment reaches UI changes before backfill, the admin UI should show
safe placeholders for missing `small` or `medium` variants and public catalog
cards should avoid silently using original bytes.

Backfill commands:

```powershell
pnpm catalog:variants:backfill:local -- --dry-run
pnpm catalog:variants:backfill:local
pnpm catalog:variants:backfill:dev:dry-run
pnpm catalog:variants:backfill:dev
pnpm catalog:variants:backfill:prod:dry-run
pnpm catalog:variants:backfill:prod
```

DEV uses `SUPABASE_DEV_URL` and `SUPABASE_DEV_SERVICE_ROLE_KEY`. PROD uses
`SUPABASE_PROD_URL` and `SUPABASE_PROD_SERVICE_ROLE_KEY`. The PROD write
command includes the required `--confirm-prod` guard; direct CLI use must also
include that flag before writing to PROD.
